# dontbesilent 问题诊断体系

来源：dontbesilent 问题消解 100 例.md  
分析时间：2026-04-27 | 核心洞察：把"错问题"变成"对问题"的诊断框架——不给答案，教你用什么角度问会自动推导出答案

---

## 核心观点

这份资料的本质是：**把错问题变成对问题的过程**。每一个常见的商业困惑，背后都是一个被问错了的问题。诊断的目的不是给答案，而是教你用什么角度问，会自动推导出答案。

---

## 【通用诊断框架】

每个问题诊断都遵循同一个结构：

```
原问题（用户真实问法） 
  ↓
错在哪里（问题本身的缺陷）
  ↓
隐含假设（这个问题背后的错误前提）
  ↓
大白话改写（用可验证的实际行为重新表述）
  ↓
真正该做的事（可落地的操作步骤）
  ↓
一句话结论（记住这个视角的核心）
```

**这个框架的威力**：它不是告诉你答案，而是告诉你"答案藏在问题里"。当你学会这个诊断方式后，遇到新问题时自动可以用。

## 【商业模式类 - 10 个核心诊断】

### 问题层级
1. **最底层问题**：「这个赛道行不行」「我适不适合创业」
   - 错误假设：存在客观的「行」和「不行」  
   - 对的问题：「有没有人愿意付钱，现在」

2. **中层问题**：「我的商业模式行不行」「怎么规模化」
   - 错误假设：模式可以脱离数据被评判
   - 对的问题：「流量、线索、成交、交付、复购各个环节有没有闭环」

3. **高层问题**：「我这个项目有没有前景」
   - 错误假设：十年后的宏大叙事 > 现在的真实成交
   - 对的问题：「现在有没有真实买家和真实利润」

### 核心诊断

| 常见问题 | 问题病根 | 改写后 | 核心操作 |
|--------|--------|--------|---------|
| 赛道还能不能做 | 把赛道当因果 | 有没有人愿意为我的产品持续付钱？ | 列出买家、产品、价格、获客、转化、交付、复购，找 3 个对标 |
| 适不适合创业 | 「适合」没有定义 | 能不能用现有资源完成一个最小交易？ | 7 天卖出一个最小产品 |
| 模式行不行 | 评判模式不带数据 | 获客、转化、交付、复购和利润有没有闭环？ | 列出最近 30 天的客户、成交、毛利、成本 |
| 没钱怎么办 | 把钱当前提条件 | 有没有不花钱也能验证的最小交易？ | 做服务、咨询、代运营、撮合、预售 |
| 有资源怎么变现 | 资源 ≠ 资产 | 能帮谁解决什么问题，谁愿意付多少钱？ | 把资源翻译成买家的收益、节省或机会 |
| 怎么做高客单价 | 把价格当目标 | 有没有值得客户多付钱的问题？ | 定义高价值场景，设计交付保障和案例 |
| ToB 还是 ToC | 分类太粗 | 更容易让哪一类买家完成付款？ | 对比两个方向的获客成本、成交周期、客单价 |
| 项目有没有前景 | 远处的雾 > 眼前的客户 | 现在有没有真实买家和真实利润？ | 先做 10 个陌生成交或 3 个复购 |
| 产品还是流量 | 二选一的假设 | 缺的是可卖的东西，还是是把买家带过来的路径？ | 先最小产品，再测转化率 |
| 怎么规模化 | 在能赚钱前想规模化 | 赚钱动作能不能被重复、复制、交给别人执行？ | 把获客、转化、交付做成 SOP，计算是否增加利润 |

**关键模式**：所有商业问题都指向一个根本问题—— **有没有真实的交易在发生**。没有交易，所有宏大叙事都是幻觉。

## 【内容创作类 - 10 个核心诊断】

### 问题层级
1. **流量问题**：「内容没流量」「要不要追热点」
   - 错误假设：流量是平台欠你的
   - 对的问题：「内容有没有竞争力争夺注意力」

2. **定位问题**：「应该做什么内容」「打造个人 IP」
   - 错误假设：存在一个合理的内容定位
   - 对的问题：「想通过内容让谁产生什么行动」

3. **输出问题**：「标题怎么写」「为什么没人看」「是否太 AI 味」
   - 错误假设：技巧能救烂内容
   - 对的问题：「有没有具体对象、冲突、判断、例子」

### 核心诊断

| 常见问题 | 问题病根 | 改写后 | 核心操作 |
|--------|--------|--------|---------|
| 内容没流量 | 找错原因 | 用户为什么要在信息流里停下来看完？ | 拆 20 条爆款，对比选题、标题、开头 |
| 应该做什么内容 | 「应该」没标准 | 想通过内容让谁产生什么行动？ | 先定买家和产品，倒推选题 |
| AI 味太重 | 错怪 AI | 文字是否空、假、滑、套、没有具体判断？ | 删套话，补对象、冲突、判断、例子 |
| 要不要追热点 | 热点不是目的 | 能不能让我说出和业务有关的判断？ | 只追能连接产品、方法论、买家的热点 |
| 短视频还是公众号 | 平台是决定 | 内容更适合快速感知还是深度阅读？ | 观点冲突→短视频，深逻辑→公众号 |
| 标题怎么写 | 技巧救不了烂选题 | 这条内容给用户什么强承诺？ | 先改选题有冲突、有收益、有对象 |
| 想做干货内容 | 「干货」是伪概念 | 能不能让用户少走一步错路？ | 用案例、标准、反例、行动替代信息堆砌 |
| 内容很专业没人看 | 专业 ≠ 阅读理由 | 用户看完能不能理解这和他的关系？ | 翻译成场景、痛点、损失、收益 |
| 怎么定位个人 IP | IP 是被动形成的 | 能长期用什么判断帮助哪类人？ | 连续输出 30 条同类高质量判断 |
| 每天发为啥没涨粉 | 频率 ≠ 因果 | 每天发的东西有没有每天变得更值得看？ | 建立复盘表：选题、开头、完播、转粉原因 |

**关键模式**：内容的本质不是表达自己，是**改变用户的下一步行动**。没有行动改变，就没有内容价值。

## 【AI 工具与 Agent 类 - 10 个核心诊断】

### 问题层级
1. **方向问题**：「怎么变现」「选什么方向」
   - 错误假设：从技术能力或时髦出发
   - 对的问题：「现在有没有人正在为某个低效流程付钱」

2. **产品问题**：「Agent 为什么没人用」「需要多强的模型」
   - 错误假设：功能强就会有人用
   - 对的问题：「有没有一个用户必须完成的、不得不交给工具的任务」

3. **系统问题**：「知识库怎么建」「怎么做 Skill」「全自动赚钱」
   - 错误假设：系统越复杂越好
   - 对的问题：「有没有可落地的工作流和可验证的结果」

### 核心诊断

| 常见问题 | 问题病根 | 改写后 | 核心操作 |
|--------|--------|--------|---------|
| 怎么变现 Agent | Agent 是形式不是产品 | 替谁完成什么高频、高价值、可验证的任务？ | 找人工服务中重复、昂贵、规则明确的流程 |
| 学提示词还是编程 | 工具 ≠ 方向 | 要自动化的任务需要语言控制还是系统集成？ | 先定义工作流，再判断需要什么工具 |
| AI 会替代我吗 | 职业层面太大 | 工作里哪些任务可以被 AI 更便宜、快、稳定地完成？ | 拆成任务清单，标注判断、沟通、执行 |
| 选什么 AI 工具方向 | 从技术出发 | 哪类人现在正在为某个低效流程付钱？ | 找 10 个正购买服务/SaaS/课程的人，问最痛点 |
| Agent 为什么没人用 | 自动化 ≠ 有用 | 用户现在有没有一个必须交给工具的任务？ | 观察用户现在怎么完成这个任务 |
| AI 写作还能做吗 | 「AI 写作」太粗 | 哪种写作任务有明确交付标准和付费意愿？ | 选垂直场景，做模板、审核标准、案例库 |
| 需要做知识库吗 | 知识库 ≠ 收藏夹 | 材料能不能帮 AI 在具体任务中做更好判断？ | 围绕任务建立案例、规则、反例 |
| 怎么把经验做 Skill | 经验不能直接变 Skill | 做这件事时按什么顺序判断，信号如何处理？ | 整理触发条件、判断流程、输出格式、禁区 |
| 需要多强的模型 | 模型强 ≠ 产品强 | 这个任务允许多大错误，谁负责，怎么纠正？ | 定义任务难度、验收标准、失败成本 |
| 能做全自动赚钱机器吗 | 把商业幻想成自动化 | 能用 AI 降低某个赚钱流程的成本或提高转化？ | 找已有收入的流程，用 AI 替换其中一段 |

**关键模式**：AI 不是魔法，是**商业流程中的一个环节**。AI 能放大已有的商业模式，但不能替你发明商业模式。

## 【知识付费与运营类 - 最后 30 个】

核心原则贯通所有知识付费问题：

```
承诺 ← → 交付 = 要不要割韭菜
定价 ← → 价值 = 多少钱合理
内容 ← → 用户转化 = 什么样的课值得上
```

关键诊断：
- **割不割韭菜**：不看品类，看有没有用假承诺收真钱
- **定价多少**：价格不是成本 + 利润，是 (用户价值 - 最便宜的替代方案) 
- **课程结构**：不是教越多越好，是教最容易有结果的部分
- **用户选择**：你的课应该筛选走 80% 的人，让能转化的 20% 留下

## 【实战应用场景】

### 场景 1：你卡在某个问题上
**步骤**：
1. 用原始问法描述你的困惑
2. 问自己：「我的隐含假设是什么」
3. 用"大白话改写"重新表述
4. 按"真正该做的事"操作

### 场景 2：你想快速判断一个想法
**步骤**：
1. 把想法套进对应的诊断框架
2. 找到"大白话改写"的那一句  
3. 立刻知道应该做什么实验

### 场景 3：你想教别人诊断
**步骤**：
1. 抛弃直接给答案的习惯
2. 按诊断框架问对方问题
3. 让对方自己得出结论

## 【这套思维的威力】

### 对标分析
相比「经验分享」和「成功学鸡汤」：
- **经验分享**：告诉你作者做对了什么 → 你用别人的解药治自己的病，通常不行
- **成功学**：告诉你「要努力」「要坚持」→ 心灵鸡汤，没用
- **这套诊断**：教你发现问题本身在哪里 → 自动推导出答案，可迁移

### 失效边界
什么时候这套诊断框架不好用：
- **当问题本身在「内部心理」而非「外部交易」时**  
  例：「我是不是该放弃」「我配不配做这件事」
  → 需要心理咨询，不是诊断框架

- **当问题牵涉其他人的决策权时**  
  例：「领导会不会同意」「客户会不会付钱」  
  → 需要沟通和谈判，不只是诊断

- **当市场环境已经完全变了时**  
  例：「这个赛道还能做吗」在行业衰退期
  → 问题本身已经不存在，问题变成了「怎么离场」

### 与其他思维体系的差异
- **vs 芒格式思维**（逆向思考）：dontbesilent 更务实，直指「有没有人付钱」；芒格式更宏观，关注「长期复利和心理偏差」
- **vs 第一性原理**：都是追问本质，但 dontbesilent 的本质是「交易是否真实存在」，第一性原理的本质是「基础物理规律」
- **vs 设计思维**：都重视用户，但 dontbesilent 直指「付费意愿」，设计思维关注「全体验」

## 【怎么用好这套框架】

### 写下来（很重要）
- 把你的问题用「原问题」形式写出来
- 尝试用「大白话改写」重写
- 列出你想要的「真正该做的事」
- 这样会自动暴露你思维的漏洞

### 问对问题（比答案重要）
- 下次有人问你「应该不应该做 X」时
- 先问他：「你现在假设了什么」
- 他自己会答出「问题在哪里」

### 定期更新（框架会进化）
- 这 100 个诊断只是起点
- 遇到新问题时，用同样的诊断框架推导
- 自己的诊断会越来越准

## 【反方视角】

这套诊断框架最可能出错的地方：

**过度简化**：不是所有问题都能用"有没有人付钱"来判断。有些问题涉及资源约束（真的没钱起步）、信息不对称（创业者不知道市场真实需求）、或长期积累（打造 IP 需要半年以上才见效）。框架会告诉你"做实验"，但实验成本可能你承受不了。

**忽视运气成分**：诊断框架假设"正确的问题 = 更好的结果"，但现实中好的创业者和差的创业者都可能问对同样的问题，结果却截然不同。这里面还有执行力、团队、市场时机、竞争格局等框架看不到的变量。

**容易变成"行动崇拜"**：框架鼓励"快速验证""7 天卖一个产品"，这在某些赛道有效，但在需要长期积累（如研发密集型产品、政府采购）的领域会彻底失效。而用这套框架的人通常会觉得"好像有人在做，我应该也能"。
